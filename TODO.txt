TASK 1 — BUG: Profile hover panel flickers (open/close loop) in game board
--------------------------------------------------------------------------------

Files involved:
  - frontend/src/app/game/components/board/board.component.ts
      (hover handlers at lines ~853-867, openProfilePanel at line ~890,
       afterRender clamp logic at line ~226)
  - frontend/src/app/game/components/board/board.component.html
      (avatar-ring at lines 16-19, profile panel + backdrop at lines 132-182)
  - frontend/src/app/game/components/board/board.component.scss
      (.profile-backdrop and .profile-panel styles — check z-index and
       pointer-events)

Reproduction:
  1. Start or join a game (route /game/...).
  2. On a desktop browser (hover-capable device), hover the mouse over any
     player's avatar (.avatar-ring) on the board.
  3. The profile panel (`.profile-panel`) opens, but immediately closes and
     reopens in a loop. The loop only stops if the cursor is moved onto the
     panel itself, which cancels the close timer via onPanelMouseEnter().

Likely root cause to investigate:
  - The panel opens at `rect.right + 8` px to the right of the avatar
    (see openProfilePanel, board.component.ts:890-905). There is an 8px gap
    between the avatar and the panel.
  - A full-viewport `.profile-backdrop` (board.component.html:133) is rendered
    behind/around the panel. When the cursor leaves `.avatar-ring`, it lands
    on the backdrop, which is neither the avatar nor the panel, so
    `onAvatarMouseLeave()` fires and starts a 150 ms close timer.
  - Additionally, the `afterRender` block (board.component.ts:226-247)
    re-clamps and re-sets `profilePanel()` after each render, which may cause
    layout thrashing and re-trigger mouseenter/mouseleave cycles.

Required fix (implement ONE coherent solution):
  a) Eliminate the dead zone between the avatar and the panel so the cursor
     can travel from avatar → panel without entering a non-tracked area.
     Options:
       - Make `.profile-backdrop` ignore pointer events while the cursor is
         intended to reach the panel (e.g. `pointer-events: none` on the
         backdrop, keeping the click-to-close handled elsewhere, OR add an
         invisible bridging hit-area between the avatar and the panel).
       - Render the panel without the full-viewport backdrop on hover-capable
         devices (keep the backdrop only when the panel was opened via click /
         touch).
  b) Ensure the `afterRender` re-positioning in board.component.ts:226-247
     does not trigger an open/close cycle (e.g. only re-clamp once on open,
     or compare position before calling `profilePanel.set(...)` to avoid
     unnecessary re-renders — already done, verify it actually short-circuits).
  c) Keep the existing 150 ms close grace timer (panelCloseTimer) but verify
     it is sufficient and that onPanelMouseEnter reliably cancels it.

Acceptance criteria:
  - Hovering the avatar opens the panel exactly once; it stays open while the
    cursor is over the avatar OR the panel (or in any bridging area between
    them).
  - Moving the cursor away from both the avatar and the panel closes the
    panel after the existing grace delay, without re-opening.
  - No visual flicker / repeated open-close cycle in any case.
  - Click-to-open behavior on touch / non-hover devices is unchanged.
  - Esc key and clicking outside still close the panel.

TASK 2 — FEATURE: Hover tooltip on disabled "Send private message" button
--------------------------------------------------------------------------------

File involved:
  - frontend/src/app/profile/profile.page.html (button at lines 53-65)
  - frontend/src/app/profile/profile.page.ts (canSendMessage() at line 83)
  - frontend/src/app/profile/profile.page.scss (.pf-send-btn at line 197)

Context:
  - On another user's profile page (/profile/:id), a "Send private message"
    button is shown when viewing someone else's profile.
  - The button is disabled when the current viewer is NOT signed in
    (canSendMessage() returns false → currentUser() is null).
  - A native `title` attribute is already set
    ("Sign in to send a message"), but native browser tooltips do not show
    reliably on disabled `<button>` elements (especially with
    `cursor: not-allowed` and `opacity: 0.5`), so guest users get no visible
    explanation of why the button is greyed out.

Required implementation:
  1. Add a visible tooltip (custom Angular/CSS tooltip, NOT only the native
     `title` attribute) that appears when a guest user hovers over the
     disabled "Send private message" button.
  2. Tooltip text (English, matching the rest of the profile page copy):
     "Sign in to send a message".
  3. The tooltip must trigger ONLY when the button is disabled
     (canSendMessage() === false). When the user is signed in and the button
     is enabled, no tooltip should be shown (or show a normal hover lift,
     unchanged from current behavior).
  4. The tooltip must be reachable by hover on a disabled element — wrap the
     button in a hover-capable container (e.g. a `<span>` with `display:
     inline-block`) so the parent receives the mouseenter events even though
     the disabled button itself may not.
  5. Style the tooltip to match the existing dark theme of the profile page
     (see .modal-card / .pf-meta styling in profile.page.scss for reference
     colors). Keep it small, with a subtle background, rounded corners, and a
     short fade-in transition (~150 ms).
  6. Accessibility: keep the existing `[title]` attribute as a fallback for
     assistive tech, and add `aria-describedby` pointing to the tooltip
     element when shown.

Acceptance criteria:
  - As a non-signed-in user visiting another user's profile, hovering the
    greyed-out "Send private message" button shows a styled tooltip with the
    text "Sign in to send a message".
  - As a signed-in user, no tooltip is shown over the enabled button.
  - On touch devices (no hover), the existing disabled state is unchanged —
    no regression.
  - No visual regression on the rest of the profile page.
