4. ROBUST GAME-STATE REFRESH WHEN RETURNING TO THE APP (MOBILE)
================================================================
Context:
- Reconnection logic lives in `frontend/src/app/app.component.ts`
  (uses `localStorage` keys `active_game_id` + `guest_player_id`,
  `TabLockService`, `GameStateService.connect/sendJoinGame`,
  `gameStarted$` / `actionRejected$`).
- The home screen shows a "You have an ongoing game — Resume →" banner
  when `localStorage.getItem('active_game_id')` and `guest_player_id`
  are both present (`home.page.ts` lines ~158-160,
  `home.page.html` lines ~80-82).
- Reported bugs on mobile:
    (a) After leaving the app overnight and reopening, the home banner
        showed "Resume the game" even though no game was actually in
        progress on the server. A hard page reload removed it.
    (b) While inside the game, switching away from the tab/app and
        coming back left the game state stale (not updated).

Goal:
- On every "app becomes visible / regains focus" event, re-validate the
  active game and the in-game state from the server, and show a clear
  loading state while we do so.

Sub-tasks:
- [x] Add a global "app resumed" listener — new `AppResumeService`
      (`services/app-resume.service.ts`, `providedIn: root`, eagerly
      created from `AppComponent`). Wires `visibilitychange` → visible,
      window `focus`, and Capacitor `App` `appStateChange { isActive }`
      once; a `wasBackgrounded` guard + 1s debounce collapses the burst
      of events into a single resume. Exposes `resumed$`.
- [x] On "app resumed" with `active_game_id` + `guest_player_id` present:
        * `validating` signal flips true → `AppComponent` shows a global
          full-screen `<app-loading-screen>` overlay (z-index 9999, also
          blocks board input during the check).
        * Re-validates via `sendJoinGame`, racing `gameStarted$` (ok) vs
          `actionRejected$` (gone) vs an 8s timeout.
        * On rejection → `removeItem('active_game_id')`, release tab lock,
          `GameStateService.reset()`, `hasActiveGame = false`.
        * On confirm → server's `gameState` message refreshes
          `GameStateService.data`; `hasActiveGame = true`.
        * The `.loading-screen` markup was extracted into a shared
          `LoadingScreenComponent` (`shared/loading-screen.component.*`),
          now used by both the game page and the global overlay.
- [x] Home page: dropped the local `activeGame` flag; the "Resume" banner
      binds to `appResume.hasActiveGame()` (a signal). `ngOnInit` calls
      `refreshFromStorage()`, and resume validation updates the signal —
      so the banner only stays once the server confirms the game.
- [x] Game page on resume: the same `AppResumeService` flow covers it —
      if `!isConnected()` it reconnects via `GameStateService.connect`
      then re-issues `sendJoinGame`; the visible state is only replaced
      when the fresh `gameState` arrives (the `data` signal). No per-page
      duplication.
- [x] Telemetry: `console.log('[AppResume] ...')` traces across the whole
      resume + validation flow.
- [~] NOT yet verified interactively (backgrounding a real device /
      overnight-stale-game repro); `ng build` passes.

5. POINTS GAIN/LOSS WEIGHTED BY OPPONENT RANKING
================================================================
Context:
- Current logic is flat: winner gets +3, every other human player gets
  -1, and a disconnect costs -2 (see
  `backend/src/game/game.ts` → `applyEndGamePoints()` at line ~757,
  and `markDisconnected()` at line ~152).
- Rankings are recomputed globally after each update via
  `recomputeRankings()` in `backend/src/db.ts`.

Goal:
- Make the points delta depend on the opponents' standing: beating
  high-ranked players should yield more points than beating low-ranked
  ones, and losing to low-ranked players should cost more than losing
  to top players.

Sub-tasks:
- [ ] Decide on a formula. Suggested simple approach (Elo-like):
        * For each non-penalized human in the game, fetch their current
          `points` (and/or `ranking`) at game end.
        * Compute per-player delta as a function of the average
          opponent rating relative to the player's own rating.
        * Keep total winner gain in a reasonable bounded range (e.g.
          +1 to +6) and loser loss in (e.g. -3 to -1) to avoid wild
          swings.
- [ ] Update `applyEndGamePoints(winnerColor)` in
      `backend/src/game/game.ts`:
        * Read current points for every human player with a `userId`
          (batch query Cosmos `users` container by id).
        * Compute deltas using the chosen formula.
        * Apply deltas via `updateUserPoints`, then
          `recomputeRankings()` once.
- [ ] Decide whether the disconnect penalty (currently flat -2 in
      `markDisconnected`) should also become rating-weighted. If yes,
      apply the same logic; if no, document the decision in a code
      comment.
- [ ] Add the per-player `pointsDelta` to the `gameEnded` message
      payload so the frontend can display it (see task 6).
- [ ] Add unit tests around the new computation (pure function on
      points/ranking input → deltas).
