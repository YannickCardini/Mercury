1. Tutorial hints wrongly re-shown after reconnecting into a game in progress
────────────────────────────────────────────────────────────────────────
Symptom:
  Close the app mid-game, then reconnect into the same game. The tutorial
  overlay shows the first-move hint "Play a King, Ace or Joker to start"
  (and the rest of the hint flow) even though the player already has
  marbles on the board, which makes the instructions incorrect.

Root cause:
  frontend/src/app/game/components/tutorial-overlay/tutorial-overlay.component.ts
  The flags `hasConfirmed` / `hasDiscarded` (lines 87-89) live only in
  component memory. They are set when the local player's `actionPlayed$`
  echo arrives, so after a reconnection (component re-created, no echo
  replayed) they start back at false and the whole hint state machine
  (`hint` computed, lines 103-126) restarts from the first step.

Fix:
  - On (re)joining a game, detect that the game is already in progress for
    the local player and permanently suppress the first-move hint flow.
  - "Already in progress" signal available in GameStateService
    (frontend/src/app/game/services/game-state.service.ts): the local
    player's `marblePositions` in `data().gameState.players` contains at
    least one marble outside its home/start zone (use the board helpers in
    packages/shared/src/board-config.ts to classify positions), or any
    other reliable indicator that the player has already played
    (e.g. cards already missing from a full hand / move history if available).
  - When that is the case, initialize `hasConfirmed` to true (and
    `hasDiscarded` if appropriate) so no stale "start" guidance appears.
  - Keep the hints working normally for a genuinely new game, including
    after a mid-first-turn refresh where the player truly has not moved yet.

2. Returning long after a game ended: ghost reconnection drops a signed-in
   user into a local guest game vs 3 hardcoded AIs
────────────────────────────────────────────────────────────────────────
Symptom:
  Quit a game, come back much later (the game has finished and no longer
  exists on the server). The app tries to rejoin the dead game, then ends
  up starting a brand-new game where the user plays as a guest (although
  signed in) against 3 local AIs — the hardcoded backend AIs, NOT the real
  matchmaking bots provided by the external AI-agent service (the ones
  obtained via the HTTP call described in README "Agent IA externe").

Root cause / mechanism (verify while fixing):
  - frontend/src/app/game/game.page.ts `connect()` (lines 151-161): if
    `active_game_id` / `guest_player_id` are missing from localStorage when
    the game page connects, it falls through to `sendStart()` (lines
    189-203), which starts a single-device debug game with the hardcoded
    config `{ Moi / IA Vert / IA Bleu / IA Orange }` via the backend
    `start` message (backend/src/index.ts case 'start' →
    sessionManager.startSingleDevice).
  - The stale-session cleanup paths clear the keys inconsistently:
    app.component.ts (line 77) removes only `active_game_id` on
    'Session expired or not found'; game.page.ts `handleLoadFailure`
    removes both; game.guard.ts requires both. A partial clear or a race
    between the server-side lookup (ActiveGameService.fetch →
    GET /api/active-game), the app-resume validation
    (frontend/src/app/services/app-resume.service.ts) and navigation to
    /game can land the user on the game page with no valid session, which
    then silently calls `sendStart()`.

Fix:
  - Remove (or strictly gate behind a debug flag) the `sendStart()`
    fallback in game.page.ts: navigating to /game without a valid
    reconnection session must NEVER silently create a debug game vs
    hardcoded AIs. Instead show the load-error path and redirect to /home
    (reuse `handleLoadFailure`).
  - Make stale-session cleanup atomic and consistent: every place that
    invalidates the session must remove BOTH `active_game_id` and
    `guest_player_id` (app.component.ts currently removes only one).
  - For signed-in users, trust the server answer: when
    GET /api/active-game returns null (no running game), clear local keys
    and stay on /home — do not attempt joinGame.
  - After the fix, verify the full scenario: signed-in user, finished
    game, app reopened → lands on /home, no guest game, no debug AIs.

3. Card-help popover lifecycle issues
   (frontend/src/app/game/components/table/table.component.ts)
────────────────────────────────────────────────────────────────────────
Context:
  Selecting a card on touch devices opens the card-effect help popover
  (`openCardHelp`, line 535) with a 3 500 ms auto-close timer
  (calls at lines 443 and 477). The popover state is `cardHelp` (line 497);
  it is closed by `closeCardHelp()` (line 565).

3a. Popover must close immediately when the card is played
  Symptom: select a card (popover appears — correct), then confirm/play the
  card: the popover stays visible until the 3.5 s timer fires instead of
  disappearing at the moment the card is played.
  Fix: call `closeCardHelp()` at the start of `onConfirmOrDiscard()`
  (line 572), so both the confirm-move and discard flows dismiss the
  popover right away. Also consider closing it when the server echoes the
  played action (`actionPlayed$`) in case the action is triggered by any
  other path.

3b. Tapping the same card again while it is NOT my turn must dismiss the
    popover (toggle behaviour)
  Symptom: when it's not my turn, tapping a card shows its help popover
  (correct, by design — see comment lines 434-445). Tapping the SAME card
  again does nothing visible: the popover stays. This only happens when it
  is not my turn.
  Root cause: in the not-my-turn branch of `onCardSelected` (lines
  434-446) the card is never actually selected (`selectedCardIndex` is not
  set), so there is no "already selected → deselect" path. The handler
  calls `closeCardHelp()` then unconditionally re-opens the popover for
  the tapped index, so a second tap just restarts it.
  Fix: track which card index the popover is currently open for (e.g. add
  the index to the `cardHelp` signal or a companion field). In the
  not-my-turn branch: if the popover is already open for the tapped index,
  close it and return (toggle); otherwise open it as today. Keep the
  desktop hover behaviour (`onCardHover` / `onCardHoverLeave`) unchanged.
