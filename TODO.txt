 2. Sign-in: add loading state during backend verification
────────────────────────────────────────────────────────────────

Context:
  - On web, `GoogleSignIn.signIn()` (called in `auth.service.ts`) redirects
    the page to Google and returns. When Google redirects back, the token
    is in the URL hash and `handleRedirectCallback()` is called
    automatically in the `AuthService` constructor.
  - `handleRedirectCallback()` posts the idToken to
    `POST /api/auth/google` — this HTTP call can take a noticeable amount
    of time.
  - On native (Android), `GoogleSignIn.signIn()` returns an idToken
    directly (no page redirect), but the subsequent backend verification
    call in `handleRedirectCallback()` still takes time.
  - Currently there is no loading indicator during this backend round-trip.

What to do:
  1. Add a `isLoading` signal (or BehaviorSubject) to `auth.service.ts`
     that is set to `true` at the start of `handleRedirectCallback()` and
     back to `false` when it completes (success or error).
  2. In `home.page.html`, display a spinner/loading indicator on the
     Google Sign-In button (or modal area) while `auth.isLoading` is true.
     Disable the button during loading to prevent duplicate requests.

 5. Bug: double timer instance causing 2× speed and duplicate sounds
────────────────────────────────────────────────────────────────

Context:
  - The turn timer (30-second countdown) is managed in `TableComponent`
    (`table.component.ts`) via a `setInterval` in `startTimer()`.
  - `startTimer()` is called from an `ngOnInit` subscription to
    `gameStateService.newTurn` (a `BehaviorSubject`).
  - Two bugs cause two simultaneous timer instances:

    Bug A — Subscription leak:
      In `ngOnInit`, the `newTurn.subscribe(...)` call is not assigned
      to a variable and is therefore never unsubscribed in `ngOnDestroy`.
      If the `TableComponent` is destroyed and recreated (e.g. by Angular
      re-rendering the game page), the old subscription remains active on
      the BehaviorSubject alongside the new one. Both subscriptions call
      `startTimer()` independently, resulting in two `setInterval` loops
      running simultaneously. This makes the timer decrement 2 seconds
      per second (turn ends in 15s instead of 30s) and causes all
      timer-triggered effects (countdown tick sounds, time-up sound,
      `sendTurnTimeout()`) to fire twice.

    Bug B — BehaviorSubject replay on subscribe:
      `newTurn` is a `BehaviorSubject` (not a plain `Subject`). On each
      new subscription it immediately emits its last stored value. If
      `TableComponent` subscribes after a turn has already started (e.g.
      on reconnect or re-render), `startTimer()` is called once on
      subscription (from the replayed value) and again when the next
      real new-turn event fires.

  Symptoms:
    - Timer counts down at double speed (2 seconds lost per real second).
    - Countdown tick sound (< 5 s remaining) plays twice per tick.
    - Time-up sound and `sendTurnTimeout()` trigger twice.
    - Bot auto-play sound fires twice when the server auto-plays.

What to do:
  1. In `table.component.ts` (`ngOnInit`), store the subscription from
     `gameStateService.newTurn.subscribe(...)` in a class property
     (e.g. `private newTurnSub?: Subscription`) and unsubscribe it in
     `ngOnDestroy`, just like `timeoutSub` and `autoPlaySub` are handled.
  2. To prevent the BehaviorSubject's initial replay from triggering an
     unwanted `startTimer()` call on subscribe, apply `.pipe(skip(1))`
     before subscribing (skip the replayed current value, only react to
     genuinely new turn events emitted after subscription), OR change
     `newTurn` from a `BehaviorSubject` to a plain `Subject` in
     `game-state.service.ts` if there is no other consumer that relies
     on the replay behavior.

 6. Update player points in the database after each game outcome
--------------------------------------------------------------------------------

When a connected (non-guest) player's game ends, update their `points` field in
Cosmos DB. Apply the following point deltas:
  - Win:    +3
  - Loss:   -1
  - Abandon (disconnect while game is in progress): -2

Files:
  - backend/src/db.ts
  - backend/src/game/game.ts
  - backend/src/session/session-manager.ts
  - backend/src/auth/auth-router.ts  (if a dedicated endpoint is needed)

Steps:
  1. In db.ts, add a function:
       updateUserPoints(userId: string, delta: number): Promise<void>
     that increments the `points` field of the user document with the given
     userId (use Cosmos DB's patch or replace operation with $inc or equivalent).

  2. After updating points, call recomputeRankings() (see task 7) to keep the
     ranking field current.

  3. In game.ts, when the game ends (first player gets all marbles to arrival):
       - Identify which players are authenticated (have a Google userId, not a
         guest ID).
       - Call updateUserPoints(userId, +3) for the winner.
       - Call updateUserPoints(userId, -1) for each losing authenticated player.

  4. In session-manager.ts (or game.ts), detect when a WebSocket closes while
     the game is still in progress (player disconnects without game ending):
       - If the disconnected player is authenticated, call
         updateUserPoints(userId, -2).
       - Do not apply this penalty if the game had already ended naturally.

  5. Guest players (no Google account) are unaffected — do not attempt any DB
     update for them.

 8. Implement the ranking (leaderboard ladder) system
--------------------------------------------------------------------------------

A player's ranking is their integer position in the global leaderboard, ordered
by points descending. The player with the most points is ranked #1. Players with
equal points share the same rank (dense ranking).

The `ranking` field already exists on every user document in Cosmos DB.

Files:
  - backend/src/db.ts
  - backend/src/auth/auth-router.ts

Steps:
  1. In db.ts, implement:
       recomputeRankings(): Promise<void>
     - Fetch all users from the `users` collection, ordered by `points` DESC.
     - Assign integer ranks starting at 1 (use dense ranking: users with equal
       points get the same rank, the next rank is the next integer, not skipped).
     - Batch-update each user's `ranking` field in Cosmos DB.
     - This function must be called after every updateUserPoints() call (task 5).

  2. In auth-router.ts, add:
       GET /api/leaderboard
     Returns an array of users sorted by ranking ASC:
       [{ name, picture, points, ranking }, ...]
     Limit to the top N players (e.g., top 100) if the user base grows large.

  3. Ensure the user's current ranking is returned as part of the auth response
     (POST /api/auth/google response body) and/or via a GET /api/user/me
     endpoint so the frontend can display it in the profile panel (task 6).
