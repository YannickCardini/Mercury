--------------------------------------------------------------------------------
TASK 1 — "Update available" prompt for the Android app
--------------------------------------------------------------------------------

GOAL
  When a new version of the app has been published, an Android user must see a
  popup as soon as they open (or resume) the mobile app, telling them a newer
  version exists and inviting them to update. The popup must offer a button that
  opens the app's Google Play Store page.

CONTEXT (current state of the codebase)
  - Frontend: Ionic + Angular (standalone components), wrapped with Capacitor 8
    for Android. appId = "online.mercury.game" (published on Google Play).
  - Installed-version source of truth on device:
      frontend/android/app/build.gradle -> versionCode 6, versionName "1.5".
    These can be read at runtime via @capacitor/app App.getInfo()
    (returns { version, build, ... }), which is already a dependency.
  - There is currently NO source of truth for "latest available version" and NO
    version-check logic anywhere.
  - App open / resume is already centralised:
      frontend/src/app/services/app-resume.service.ts listens to
      App.addListener('appStateChange') and visibilitychange. Cold start is
      handled in frontend/src/app/app.component.ts ngOnInit.
  - Modal convention (do NOT use Ionic ModalController): a standalone component
    with `@Input() show` and `@Output() closeModal`, rendered conditionally in a
    parent template. Reference: frontend/src/app/shared/game-rules-modal.component.ts
    (used from game.page.html / home.page.html).
  - HTTP is available: provideHttpClient() is set in frontend/src/main.ts;
    HttpClient is injected in services such as auth.service.ts.
  - Backend: Express, mounts REST routers at /api/auth and /api/messages
    (see backend/src/index.ts). Adding a new public GET endpoint is trivial.
  - apiUrl is defined per-environment in
    frontend/src/environments/environment(.prod).ts.

DECISION 
  "Latest available version" will be served by the OUR OWN BACKEND, not scraped
  from the Play Store. This keeps the check simple, offline-fail-safe, and lets
  us flag a version as a *forced* update later if needed.

SUB-TASKS

  Backend
    [ ] Add a public endpoint `GET /api/version` (no auth) in
        backend/src/index.ts (or a small dedicated router file
        backend/src/version/version-router.ts mounted like the others).
    [ ] Response shape (JSON):
            {
              "latestVersionCode": 6,      // integer, matches build.gradle versionCode
              "latestVersionName": "1.5",  // human-readable, matches versionName
              "minVersionCode": 1,         // below this => force update (optional, future)
              "storeUrl": "https://play.google.com/store/apps/details?id=online.mercury.game"
            }
    [ ] Source the values from an env var / config constant so a new release can
        bump the number without a code change. Document where to update it on
        each release (this value must be raised whenever build.gradle versionCode
        is raised).

  Frontend — version check service
    [ ] Create a new service, e.g.
        frontend/src/app/services/app-update.service.ts (providedIn: 'root').
    [ ] On check, do nothing unless Capacitor.isNativePlatform() is true AND the
        platform is Android (Capacitor.getPlatform() === 'android'). Web users
        must never see the popup.
    [ ] Read the installed build number via App.getInfo() (use `build`, which
        maps to Android versionCode).
    [ ] Fetch GET {environment.apiUrl}/api/version via HttpClient.
    [ ] Compare installed `build` vs `latestVersionCode`. If installed is lower,
        flag "update available" (expose a signal, e.g. updateAvailable()).
    [ ] Fail silently on any network/error (never block app usage, never throw).
    [ ] Optional: debounce so the network check runs at most once per app
        foregrounding, not on every visibilitychange burst.

  Frontend — wiring the check to app open/resume
    [ ] Run the check on cold start (app.component.ts ngOnInit) and on resume
        (subscribe to AppResumeService.resumed$ in app-resume.service.ts, or
        trigger from the same place appStateChange is handled). Avoid wiring a
        listener per page.

  Frontend — the popup UI
    [ ] Create a standalone modal component following the game-rules-modal
        convention, e.g.
        frontend/src/app/shared/update-available-modal.component.{ts,html,scss}
        with `@Input() show` and `@Output() closeModal`.
    [ ] Content: short title + message ("A new version is available, please
        update the app"), a primary button "Update" that opens `storeUrl`, and
        (for a non-forced update) a secondary "Later" / dismiss action.
    [ ] Open the store link with @capacitor/app App or window.open(storeUrl,
        '_system') so it leaves the WebView and lands on the Play Store app/page.
    [ ] Render the modal once at app level (app.component.html) bound to
        AppUpdateService.updateAvailable(), so it can appear over any page on
        open/resume.
    [ ] Match existing modal styling (reuse the look & feel / overlay of
        game-rules-modal.component.scss).

  Texts / i18n
    [ ] Wording: provide French text to match the existing UI language (the app's
        copy and code comments are predominantly French).

  The update should be mandatory

ACCEPTANCE CRITERIA
  - On an Android device whose installed versionCode < latestVersionCode, opening
    or resuming the app shows the update popup; tapping "Update" opens the Play
    Store page for online.mercury.game.
  - When installed versionCode >= latestVersionCode, no popup appears.
  - On web (and iOS, if ever built), the popup never appears.
  - If /api/version is unreachable, the app behaves exactly as before (no popup,
    no error surfaced to the user).
