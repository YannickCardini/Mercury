# TODO

## 1. Server-authoritative game reconnection for signed-in users

### Bug report (original incident)
An Android player's phone shut down (dead battery) mid-game. The bot correctly
took over his turns (expected: 180 s reconnect window, then bot plays). After
restarting his phone, the player — still signed in with his Google account —
landed on the home page with no way to rejoin the ongoing game, even though the
game was still running with the bot playing his color.

### Root cause (confirmed in the code)
Reconnection depends **exclusively** on two client-side localStorage keys,
`guest_player_id` and `active_game_id`:
- `frontend/src/app/app.component.ts` (ngOnInit, ~line 45): reconnects only if
  both keys are present, via the `joinGame` WS message.
- `frontend/src/app/game/game.guard.ts`: allows `/game` only if both keys exist,
  otherwise redirects to `/home`.
- Backend: `SessionManager.playerIdentities` (`backend/src/session/session-manager.ts`,
  ~line 35) maps a random `guestPlayerId` → `{ gameId, color }`. There is **no
  mapping from the authenticated `userId` to an active game** anywhere in the
  backend, and no endpoint to query it.

If localStorage is lost or cleared (WebView data loss after a hard shutdown, or
the `actionRejected` handler in `app.component.ts` ~line 64 deleting
`active_game_id` after a single transient rejection), a signed-in player has no
path back into their game.

### Required behavior
While a signed-in user is a player in an active (unfinished) game:
1. Any client authenticated with that account must be **redirected into the
   game** — it must be impossible to stay on the home page.
2. That account must be **blocked from joining or creating any other game**
   (public matchmaking queue, custom room create/join, multi-device room).
3. The only ways to be released from the game are: the game ending, or the
   player resigning via the in-game **resign** button (`abandonGame` message →
   `handleAbandonGame` in `backend/src/game/game.ts`, ~line 418).

### Implementation steps

**Backend**
- [x] Add a server-side map `userId → { gameId, color, guestPlayerId }` for
      signed-in players, populated everywhere guest identities are created:
      `SessionManager.startSingleDevice`, `SessionManager.joinRoom` (welcome
      block), `MatchmakingManager` (game start, where `guestPlayerId` is
      issued), `CustomGameManager` (game start).
      → `backend/src/session/reconnect-registry.ts` (new `ReconnectRegistry`
      with `byGuest` + `byUser` indexes); replaces the old `playerIdentities`
      Map across SessionManager / Matchmaking / CustomGame managers.
- [x] Clear the entry when the player is released: game end / abort (all the
      `GameRegistry.delete` paths in `game.ts`) and resign.
      → New `Game.setOnGameEnded` fires at every `GameRegistry.delete` site
      (`releaseGame`); `handleAbandonGame` now `forceDisconnect`s everyone and
      calls `onPlayerAbandoned` → `releaseSlot`, so resign RELEASES the account
      (rejoin no longer possible — the bot plays the color out). Dead
      `softDisconnect` / `indefiniteReconnect` removed from the messenger.
- [x] Add an authenticated lookup `GET /api/active-game` (Bearer session token),
      returning `{ gameId, guestPlayerId, color }` or `null`.
      → `backend/src/index.ts` (uses `verifyAuth`, validates the game still
      exists in `GameRegistry`).
- [x] Enforce the join block server-side: in `joinMatchmaking`,
      `createCustomRoom`, `joinCustomRoom`, `createRoom`, if the caller's
      `userId` already maps to an active game, reject with `alreadyInActiveGame`
      (carries gameId/guestPlayerId/color so the client can redirect in).
      → `SessionManager.rejectIfInActiveGame` (also self-heals stale entries).
      `joinRoom` carries no userId in its message, so it can't be blocked there.

**Frontend**
- [x] On app startup (`app.component.ts` ngOnInit) and on home page entry, when
      signed in, call `GET /api/active-game` and reconcile localStorage against
      it (server is the source of truth — recovers wiped keys, clears stale
      ones), then reconnect / redirect to `/game`.
      → New `services/active-game.service.ts`; `app.component.syncSignedInActiveGame`
      + `home.page.redirectIfServerHasActiveGame`.
- [x] Signed-in user with a server-reported active game cannot remain on
      `/home` — `home.page` redirects to `/game`.
- [x] Handle the `alreadyInActiveGame` rejection on matchmaking / custom-room
      joins by reconnecting and redirecting into the game.
      → `game-state.service` restores localStorage + emits `alreadyInActiveGame$`;
      `home.page` subscribes and reconnects via `joinGame`.
- [x] Make session-drop less aggressive: the cold-start `actionRejected`
      handler now only drops `active_game_id` on the explicit
      "Session expired or not found" reason, not on transient errors.

### Acceptance criteria
- Signed in, in a matchmaking game: kill the app and wipe app data (simulates
  the dead-battery localStorage loss) → reopen the app, sign in → automatically
  redirected into the still-running game with full state restored.
- While that game is active: navigating to `/home` (any device, same account)
  redirects to `/game`; joining matcrth flagging
joinRoom (local multi-devhmaking or a custom room is rejected and
  redirects to `/game`.
- Pressing the in-game resign button releases the account: home page is
  accessible and new games can be joined.
- Guest (not signed-in) players keep the current localStorage-based
  reconnection behavior unchanged.
