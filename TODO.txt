1) BUG — Invincibilité incorrecte d'un pion qui revient sur sa case de départ
================================================================================

Problème observé
----------------
Un adversaire a utilisé un J pour échanger son pion avec le mien, ce qui m'a
posé sur MA propre case de départ. Ensuite je n'ai pas pu jouer un 4 pour
reculer ce pion alors que j'aurais dû en avoir le droit.

Règle correcte
--------------
Un pion n'est "invincible" QUE pendant la fenêtre suivante :
  - Il vient d'entrer en jeu via un As ou un Roi sur sa case de départ.
  - Il n'a jamais bougé depuis cette entrée.

Dès qu'il bouge une fois, il n'est plus jamais invincible — sauf s'il meurt
(retour à la maison) puis ré-entre via un As ou un Roi. Le simple fait de
re-passer ou ré-atterrir sur sa case de départ (via un 4, un J, ou tout autre
moyen) ne redonne PAS l'invincibilité.

Conséquences sur les règles :
  - Un pion qui n'est plus invincible mais qui se trouve sur sa case de départ
    PEUT être reculé avec un 4.
  - Un pion qui n'est plus invincible mais qui se trouve sur sa case de départ
    PEUT être échangé avec un J.
  - Un pion qui n'est plus invincible mais qui se trouve sur sa case de départ
    NE BLOQUE PAS le chemin des autres pions (n'importe quelle couleur).

Travail à faire
---------------
a) Modèle de données — ajouter un état "invincible" par pion :
   - Étendre le modèle de pion (probablement dans backend/src/game/player.ts
     et/ou packages/shared/src/types.ts) avec un flag par pion, p. ex.
     `marbleInvincible: boolean[]` indexé comme `marblePositions`.
   - Initialisation : `false` pour les pions en maison.
   - Mise à jour :
       * Entrée en jeu via As ou Roi (Action de type 'enter')  → flag passe à `true`.
       * Le pion bouge (Action 'move', 'capture', 'swap', 'promote', recul par 4,
         déplacement issu d'un split de 7, etc.) → flag passe à `false`.
       * Le pion est capturé / retourne en maison → flag remis à `false`.

b) Validateur (packages/shared/src/move-validator.ts) — remplacer toutes les
   conditions "pion sur sa case de départ" par "pion invincible" :
     * `buildBackwardMoveAction` (carte 4) :
         - Aujourd'hui, l'interdiction de reculer depuis la propre case de
           départ est ligne 146 (`if (START_POSITIONS[playerColor] === marblePosition) return null;`).
           À supprimer ou remplacer par une vérification du flag d'invincibilité
           du pion qui recule (un pion non-invincible peut reculer même
           s'il se trouve sur sa case de départ).
         - La boucle ligne 227-232 qui rejette les pions adverses sur leur propre
           start doit utiliser le flag d'invincibilité, pas la position seule.
     * `pathIsClear` (lignes 273-304) : un pion adverse sur SA propre case de

     * Cas du J (lignes 122-142) : `swappableTargets` exclut aujourd'hui les
       pions assis sur leur propre start (lignes 130-131). Remplacer par
       "exclut les pions invincibles".
     * Toute autre référence à `getStartPositionOwner` / `isAnyStartPosition`
       qui sert à détecter de l'invincibilité doit consulter le flag.

c) Propager le flag dans le `LegalMoveContext` :
   - Ajouter par exemple `invincibleMarblesByColor: Record<MarbleColor, number[]>`
     ou `invincibleMarbles: number[]` (positions courantes des pions invincibles).
   - Mettre à jour tous les appelants (backend/src/game/game.ts,

d) Sérialisation réseau :

e) Tests à écrire / faire passer :

   - Mon pion sort de mon start puis y revient (par 4 / J / déplacement) →

   - Mon pion est capturé puis ré-entre via As/K → il redevient invincible.

2) BUG — Validation incomplète du 7 (split) avec promotion
================================================================================

Problème observé
----------------
Un joueur vert a tenté de jouer un 7 réparti ainsi :
  - 6 pas pour pion A → entrée sur la première case d'arrivée verte.
  - 1 pas pour pion B → bloqué par la case de départ du joueur bleu occupée
    par un pion bleu.
Le validateur a refusé le 7 ET en plus le joueur N'A PAS pu défausser le 7
(le bouton "Discard" n'apparaissait pas). Il s'est retrouvé bloqué.

Règle correcte
--------------
Pour un split de 7 :
  - Le premier mouvement modifie l'état du plateau (allMarbles, marblesByColor,
    et la liste des cases d'arrivée encore libres pour le joueur).
  - Le second mouvement doit être validé contre cet état mis à jour.
  - Si le second pion est de la même couleur que le premier ET vise la même
    case d'arrivée que celle prédite pour le premier, c'est ILLÉGAL
    (deux pions de la même couleur sur la même case).
  - Si AUCUN split (1+6, 2+5, ... 6+1) ni AUCUN mouvement plein (7 d'un seul
    pion) n'est légal, le joueur doit pouvoir défausser le 7 (`canDiscard`
    doit être true).

Travail à faire
---------------
a) Audit de `getLegalSplit7Action` (move-validator.ts:357) :
   - Vérifier que ctx2 reflète bien le résultat du premier mouvement quand
     celui-ci est une promotion (action1.type === 'promote'). Aujourd'hui
     `to1 = action1.to` est utilisé comme remplacement de `from1` dans
     allMarbles / ownMarbles / marblesByColor. Confirmer que cela injecte
     correctement la nouvelle position d'arrivée et que `getArrivelCaseIfCanPromote`
     appelé pour le 2e pion la voit comme occupée (et l'exclut donc des cases
     d'arrivée libres).
   - Cas particulier : si le 2e pion vise aussi une case d'arrivée (promote +
     promote), s'assurer que la case calculée pour lui n'est pas celle déjà
     occupée par le 1er.

b) Audit de la détection "aucun coup légal" (calcul de `canDiscard`) :
   - Trouver où le serveur calcule `canDiscard` (probablement dans
     backend/src/game/game.ts ou human-strategy.ts ; chercher les références à
     `canDiscard` et `findLegalMoveForCard`).
   - Pour la carte 7, il faut considérer comme légal :
       * tout 7-entier valide sur un pion (steps = 7), ET
       * tout split (steps1 in 1..6) tel qu'il existe au moins un second pion
         (différent du premier) pouvant avancer de (7 - steps1).
   - Si AUCUNE de ces options ne renvoie une Action non-nulle pour AUCUNE
     paire de pions, alors le 7 est défaussable.

c) Audit du frontend (table.component.ts:139 et suivants) :
   - `validSplitSevenSteps()` calcule les steps1 légaux pour le pion
     sélectionné. Vérifier que pour chaque steps1, on cherche au moins un
     autre pion pouvant absorber (7 - steps1). C'est déjà fait (ligne 152-159)
     mais ré-éprouver le cas où le seul second pion candidat finit sur la
     même case d'arrivée que le premier.
   - Vérifier le bouton "Discard" : il doit s'activer dès que `canDiscard` est
     true côté serveur, même si la carte 7 est sélectionnée.

d) Tests à écrire :
   - Split 6+1 où le 1+ est bloqué par une case de départ adverse occupée par
     un pion invincible → split refusé.
   - Split 6+1 où le 2e pion vise la même case d'arrivée que le 1er → split
     refusé.
   - Aucun split possible + aucun 7-entier possible → `canDiscard === true`
     ET bouton Discard activé côté frontend.
   - Cas du joueur vert (reproduire la partie observée) : le serveur doit
     proposer la défausse.

3) UX — Persistance des notifications d'invitation à une partie custom
================================================================================

Problème observé
----------------
Les notifications d'invitation (`gameInvite`) s'évaporent dans plusieurs cas :
  - L'utilisateur ne réagit pas assez vite (auto-dismiss au bout de 15 s dans
    frontend/src/app/shared/invite-toast.component.ts:5).
  - L'utilisateur n'est pas connecté à la WebSocket de présence au moment de
    l'invitation et la perd entièrement.
  - L'utilisateur n'est pas sur la page d'accueil au moment où la notification
    arrive — elle est droppée (home.page.ts:191-195 : la souscription
    `gameInvite$` n'existe que dans `ngOnInit` de la HomePage).
  - L'utilisateur quitte l'app (background / switch d'app sur mobile) → la
    WebSocket de présence se ferme, l'invitation reçue avant est perdue,
    et la dialog de custom game se referme (perte du flux pour TOUS les
    joueurs déjà dans la room).

Comportement attendu
--------------------
Une invitation reste affichée tant que :
  - L'utilisateur invité n'a pas explicitement cliqué Join OU Decline, ET
  - L'invitant n'a pas annulé l'invitation OU fermé la custom room.

Le passage en arrière-plan, le switch d'application ou la perte du socket NE
doit PAS faire disparaître la notification, ni faire quitter la custom room
au créateur ni aux autres participants.

Travail à faire
---------------
a) Côté toast invité (frontend/src/app/shared/invite-toast.component.ts) :
   - Supprimer le timer d'auto-dismiss de 15 s (lignes 5 + 26-35) OU le
     remplacer par "aucun auto-dismiss".
   - L'invitation ne disparaît que sur clic Join, clic Decline, ou réception
     d'un signal d'annulation envoyé par le serveur (voir point d).

b) Côté souscription frontend (frontend/src/app/home/home.page.ts) :
   - L'invitation reçue alors que `showCustomGame === true` ou
     `appResume.hasActiveGame()` est aujourd'hui silencieusement droppée
     (ligne 193). Conserver le drop mais la stocker côté serveur (déjà fait via
     Cosmos `invitations` container, voir backend/src/messages/invitations-router.ts)
     pour ré-afficher la dernière en attente quand l'utilisateur revient sur
     home et est idle.
   - Note historique : `fetchPendingInvitations` (HTTP GET /api/invitations/pending)
     a été retirée — redondante avec le flush in-memory natif que
     `PresenceManager.register` effectue dès que la socket de présence s'ouvre.
     Toute invitation queued est livrée à ce moment-là.

c) Côté backend (backend/src/session/presence-manager.ts +
   backend/src/session/custom-game-manager.ts) :
   - Persistance offline 100 % in-memory via `PresenceManager.sendOrQueue`
     (TTL 5 min). PAS de Cosmos : le free-tier 1000 RU/s ne supporte pas
     un 4ᵉ container. Trade-off accepté : un redémarrage backend perd les
     invitations en file (cas rare hobby).
   - Ajouter un endpoint (ou un message WS) pour ANNULER une invitation
     côté serveur (à appeler quand le créateur ferme la custom room ou retire
     manuellement l'invité). Le serveur retire l'entrée queued via
     `PresenceManager.cancelQueued(...)` ET pousse un message
     `gameInviteCancelled` à l'invité s'il est connecté.
   - Définir un nouveau type WS `GameInviteCancelledMessage` dans
     packages/shared/src/types.ts (champs : `type: 'gameInviteCancelled'`,
     `fromUserId: string`, `roomCode: string`).

d) Persistance UI au switch d'application :
   - Dialog "custom game" actuellement détruite quand le composant Home est
     démonté ou que la WS se ferme. Identifier dans home.page.ts les
     handlers qui ferment la dialog (`cleanupCustomGame`, `cancelCustomGame`,
     `customConnectionErrorSub` à la ligne 471) et faire la distinction entre
     "fermeture volontaire de l'utilisateur" et "perte de socket par mise en
     arrière-plan".
   - À la perte de socket due à un background (détectable via
     AppResumeService.wasBackgrounded), conserver l'état UI de la room et
     tenter une reconnexion automatique au retour de l'app (resume) plutôt
     que d'afficher l'erreur de matchmaking.
   - Côté serveur, la session de custom room doit survivre à une déconnexion
     courte (grace period de reconnexion comparable aux 180 s déjà en place
     pour les parties actives, à confirmer dans
     backend/src/session/custom-game-manager.ts).

5) FEATURE — Chat / emojis animés pendant une partie
================================================================================

Objectif
--------
Permettre aux joueurs d'interagir de façon conviviale pendant la partie
(messages courts ou emojis animés type "réactions") sans encombrer l'interface
de jeu déjà dense sur mobile.

Contraintes
-----------
- La majorité des joueurs sont sur mobile, écran réduit, interface déjà bien
  remplie (plateau + main + timer + footer).
- Aucune régression sur le gameplay existant.
- Ergonomie pouce-friendly.

Piste à explorer (à valider)
----------------------------
Ajouter un système de tabs (ou de switch) AU-DESSUS du `TableComponent`
permettant de basculer entre :
  - Vue "cartes" (actuelle, par défaut),
  - Vue "chat" (nouvelle).
Le switch ne doit pas dégrader la lisibilité du plateau.

Travail préliminaire à faire (pas d'implémentation tant que le design n'est
pas validé)
--------------------------------------------------------------------------
a) Lister les options de design possibles :
     1. Tabs au-dessus du TableComponent (cartes ↔ chat), comme suggéré.
     2. Bulle "réactions" flottante en coin (emojis animés uniquement, pas de
        texte libre).
     3. Tiroir latéral / bottom-sheet ouvert via une icône (icône à intégrer
        dans `.action-menu`).
     4. Combinaison (emojis flottants tap-rapide + chat texte caché dans un
        tiroir).
   Pour chaque option, lister avantages / inconvénients UX et impact sur le
   layout actuel.

b) Décider ensemble (utilisateur + IA) :
     - Texte libre OU emojis prédéfinis OU les deux ?
     - Modération / signalement nécessaire ?
     - Historique persistant ou éphémère à la fin de la partie ?

c) Définir le contrat WebSocket :
     - Nouveau message client→serveur (p. ex.
       `ChatMessage` : `{ type: 'chat', content: string }` ou
       `EmojiReaction` : `{ type: 'reaction', emoji: string }`).
     - Nouveau message serveur→clients (broadcast à la room) avec horodatage
       et auteur (couleur + nom).
     - À placer dans packages/shared/src/types.ts.

d) NE PAS coder l'implémentation tant que les points (a), (b), (c) ne sont
   pas tranchés avec l'utilisateur.
