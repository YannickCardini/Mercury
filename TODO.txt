## 1. Android build: show the native versionName instead of the package.json version

Context:
- `frontend/src/app/home/home.page.ts` (line 9) imports `version` from the root
  `package.json` and exposes it as `appVersion` (line 63).
- `appVersion` is displayed in two places in `frontend/src/app/home/home.page.html`:
  - the home footer button: `v{{ appVersion }}` (line ~125)
  - the "Version" row of the settings modal (line ~272), which also carries the
    hidden worker-login easter egg (`onVersionTap()`, 5 taps).
- The Android app defines its own version in
  `frontend/android/app/build.gradle`: `versionCode 7`, `versionName "1.7"`.
  This is the version users actually see in the Play Store, so showing the
  package.json version (`0.12.0`) on Android is misleading.
- `@capacitor/app` (v8) is already installed in `frontend/package.json`. Its
  `App.getInfo()` returns `version` (= Android `versionName`) and `build`
  (= Android `versionCode`).

Task:
- When running as the native Android app (`Capacitor.getPlatform() === 'android'`,
  see existing usage in `frontend/src/app/services/app-update.service.ts:44`),
  display the Android `versionName` from `App.getInfo()` instead of the
  package.json version. Fall back to `versionCode` only if retrieving the
  `versionName` proves too complicated.
- On web (non-native), keep displaying the package.json version as today.
- Update both display locations (home footer and settings modal) consistently.
- Do not break the 5-tap easter egg bound to the settings "Version" row.
- Since `App.getInfo()` is async, make `appVersion` reactive (e.g. a signal
  initialized with the package.json version, then overwritten on Android once
  the native info resolves).

## 2. Mobile UX: replace long-press as the trigger for the card-help popover

Context:
- In-game, each card in the player's hand has a contextual help popover
  (title + effect description). The text comes from the single source of truth
  `getCardEffect()` in `frontend/src/app/shared/card-effects.ts` (also used by
  the rules modal — never duplicate these texts).
- Implementation lives in
  `frontend/src/app/game/components/table/table.component.ts`
  (section "Aide contextuelle sur les cartes", lines ~469-590):
  - Desktop (devices matching `(hover: hover)`): popover opens on mouse hover,
    closes on mouseleave. This works well — DO NOT change desktop behavior.
  - Mobile (touch): popover opens after a 420 ms long-press
    (`LONG_PRESS_MS`, `onCardPressStart/Move/End`), and the subsequent
    selection click is swallowed via `suppressClickIndex`.
- Problem: the long-press gesture on mobile is neither discoverable nor
  ergonomic. Users don't know it exists, and holding a card conflicts mentally
  with tapping to select it.

Task:
- Design and implement a more intuitive, discoverable way to show the card help
  on touch devices (devices NOT matching `(hover: hover)`). Candidate
  approaches (pick the most sensible, or propose better):
  - A small visible info affordance (e.g. a "?" / ⓘ icon) on each card or on
    the currently selected card that opens the popover on tap.
  - A persistent help toggle near the hand: when enabled, tapping a card shows
    its description instead of (or in addition to) selecting it.
  - Showing the help text of the selected card in a dedicated area when a card
    is selected.
- Constraints:
  - Tapping a card to select/play it must remain the primary, unimpeded action.
  - Keep using `getCardEffect()` as the only source of the help texts.
  - Keep the desktop hover behavior unchanged.
  - Once the new mobile mechanism is in place, remove the now-dead long-press
    machinery (`LONG_PRESS_MS`, `longPressTimer`, `pressStartX/Y`,
    `suppressClickIndex`, `onCardPressStart/Move/End`) if nothing else uses it.
