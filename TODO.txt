#          in-game "Confirm" / "Discard" button
# ────────────────────────────────────────────────────────────────────
#
# CONTEXT
#   On Android, when the user taps slightly imprecisely (e.g. a long
#   press or a tap-drag) on the main action button of the game board,
#   the OS triggers its native text-selection popup ("Copy / Paste /
#   Select all…") and visually selects the button label
#   ("Confirm" or "Discard"). This breaks the gameplay flow.
#
# WHERE
#   - Button markup:
#       frontend/src/app/game/components/table/table.component.html
#         lines 175–190  (.action-btn-wrap / .action-btn)
#   - Button styles:
#       frontend/src/app/game/components/table/table.component.scss
#         lines 683–722  (.action-btn — currently NO user-select rule)
#
# WHAT TO DO
#   1. In table.component.scss, on `.action-btn` (and on
#      `.action-btn-wrap` to be safe), add:
#         user-select: none;
#         -webkit-user-select: none;
#         -webkit-touch-callout: none;          /* disables the long-press menu on iOS/Android WebView */
#         touch-action: manipulation;           /* removes the 300ms double-tap delay and the long-press */
#   2. Verify the same fix is applied to the in-game "Resign" confirm
#      button (`.resign-dialog__confirm` in the same SCSS file, ~line 1294

#      imprecise tap no longer triggers the system selection popup.


# ────────────────────────────────────────────────────────────────────
# TASK 2 — Reconnection: a player with an ongoing game must always
#          be sent back to that game when reopening the app
# ────────────────────────────────────────────────────────────────────
#
# CONTEXT / OBSERVED BUG
#   A friend played a game on Android. He got disconnected several
#   times during the match (network drop, app backgrounded, etc.).
#   When he reopened the app, instead of being put back into his
#   ongoing game, he was left stuck on the home screen — and pressing
#   "Play Now" started a NEW matchmaking session, abandoning the
#   game he was still part of.
#
# GOAL (desired final behaviour)
#   A) A user disconnect (WebSocket close, app backgrounded, browser
#      tab closed, network drop, …) must NEVER, on its own, be
#      interpreted by the backend as an abandon/forfeit. The server
#      already has a 180 s reconnection window (RECONNECT_WINDOW_MS in
#      backend/src/game/game-messenger.ts) and an indefinite window for
#      signed-in players — make sure both code paths are still hit and
#      that no other path penalises the player (-2 pts, win-by-default,
#      etc.) before the window has expired.
#   B) Whenever a connected user reopens the app/website AND has an
#      active game stored locally (`localStorage.active_game_id` +
#      `localStorage.guest_player_id`), they must be AUTOMATICALLY
#      redirected to /game and reconnected to that game, no matter
#      which route they would otherwise land on.
#   C) The user must NOT be able to reach the home menu or start a
#      new matchmaking / custom game as long as the server still
#      considers them part of an ongoing game.
#   D) The only ways to leave an ongoing game must be:
#         - explicitly resigning (existing "Resign" button in the
#           game UI — `resign()` / `confirmResign()` in
#           frontend/src/app/game/components/table/table.component.ts),
#         - finishing the game normally (victory / loss).
#
# WHERE (current implementation that must be reviewed/changed)
#   Frontend:
#     - frontend/src/app/app.component.ts (lines 23–62)
#         Already reconnects on app boot if localStorage has an active
#         game AND navigates to /game on `gameStarted$`. Verify this is
#         actually triggered for a returning Android user and is not
#         short-circuited by `tabLock.isOtherTabActive()`.
#     - frontend/src/app/services/app-resume.service.ts
#         Already re-validates the active game against the server on
#         resume but DOES NOT redirect to /game on success. After
#         `gameStarted$` fires inside `validateActiveGame()` (line ~130),
#         add a `router.navigate(['/game'])` so the user is forcibly
